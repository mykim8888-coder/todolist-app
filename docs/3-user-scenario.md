# TodoListApp 사용자 시나리오 문서

> 버전: 0.1.0-draft | 작성일: 2026-05-13 | 참조: PRD v0.2.0-draft (`docs/2-prd.md`)

---

## 문서 개요

본 문서는 TodoListApp PRD에 정의된 유스케이스 UC-01~UC-12 전체를 실제 사용자 관점에서 서술한 시나리오 문서다. 각 시나리오는 가상의 페르소나를 기반으로 작성되어 있으며, 비즈니스 규칙(BR-01~BR-08) 및 비기능 요구사항(NFR-01~NFR-07)이 자연스럽게 반영되어 있다.

### 주요 페르소나

| 페르소나 | 나이 | 직업 | 특징 |
|---------|------|------|------|
| 김민영 | 32세 | IT 기업 기획자 | 업무와 개인 일정을 분리해서 관리하고 싶어함. PC와 스마트폰을 모두 사용 |
| 박준혁 | 45세 | 중견기업 팀장 | 여러 프로젝트를 병행하며 할일이 많음. 실수로 데이터 삭제가 두려움 |
| 이서연 | 26세 | 스타트업 디자이너 | 모바일 중심 사용자. 이동 중에도 빠르게 할일을 확인하고 싶어함 |

### SCN-01: 신규 회원가입

- **관련 UC**: UC-01
- **관련 규칙**: BR-01, NFR-03

**사전 조건 (Precondition)**
- 김민영은 TodoListApp에 계정이 없는 신규 사용자다.
- 서비스 접속 URL에 접근 가능한 상태이며, 현재 비인증 상태다.

**트리거 (Trigger)**
- 회사 동료로부터 TodoListApp을 추천받은 김민영이 퇴근 후 집에서 PC 브라우저로 서비스에 접속한다. 메인 페이지가 `/login`으로 리다이렉트되는 것을 보고 "회원가입" 링크를 클릭한다.

**정상 흐름 (Main Flow)**
1. 김민영은 `/signup` 페이지에서 이름(`김민영`), 이메일(`minkim@example.com`), 비밀번호(`Planning123`)를 입력 폼에 기입한다.
2. 비밀번호는 8자 이상, 영문과 숫자를 조합해야 한다는 안내 문구를 확인하고 조건에 맞게 입력한다 (NFR-03).
3. "회원가입" 버튼을 클릭한다.
4. 시스템은 `POST /api/auth/signup` 요청을 처리한다. 이메일 중복 여부를 검사하고(BR-01), 비밀번호를 bcrypt(salt rounds 12)로 해싱하여 DB에 저장한다(NFR-03).
5. 회원가입 성공 응답(201)을 받은 브라우저는 자동으로 `/login` 페이지로 이동한다.
6. 성공 안내 메시지("회원가입이 완료되었습니다. 로그인해 주세요.")가 표시된다.

**예외 흐름 (Alternative Flow)**
- **A1. 비밀번호 형식 오류**: 비밀번호가 8자 미만이거나 영문·숫자 조합을 충족하지 못하면, 폼 유효성 검사 오류 메시지가 즉시 표시되고 서버 요청은 전송되지 않는다.
- **A2. 필수 항목 미입력**: 이름, 이메일, 비밀번호 중 하나라도 비어 있으면 제출 버튼 클릭 시 해당 필드에 오류 표시가 나타난다.
- **A3. 이메일 형식 오류**: 유효하지 않은 이메일 형식 입력 시 클라이언트 단 유효성 검사에서 즉시 안내한다.
- **A4. 연속 회원가입 시도**: 분당 10회를 초과하는 회원가입 시도 발생 시 시스템은 Rate Limiting을 적용하여 429 응답을 반환하고 잠시 후 다시 시도하도록 안내한다(NFR-03).

**사후 조건 (Postcondition)**
- `users` 테이블에 김민영의 계정 레코드가 생성된다(`auth_provider = 'local'`).
- 비밀번호는 평문으로 저장되지 않고 bcrypt 해시값으로 저장된다.
- 김민영은 `/login` 페이지로 이동하여 로그인을 진행할 수 있는 상태가 된다.

### SCN-02: 이메일 중복으로 인한 회원가입 실패

- **관련 UC**: UC-01
- **관련 규칙**: BR-01

**사전 조건 (Precondition)**
- `minkim@example.com` 이메일은 이미 DB에 등록되어 있다.
- 다른 기기에서 접속한 사용자가 동일한 이메일로 가입을 시도하는 상황이다.

**트리거 (Trigger)**
- 김민영이 오래전에 가입했던 사실을 잊고, 새 브라우저에서 동일한 이메일로 회원가입을 시도한다.

**정상 흐름 (Main Flow)**
1. `/signup` 페이지에서 기존 이메일(`minkim@example.com`), 이름, 비밀번호를 입력하고 "회원가입" 버튼을 클릭한다.
2. 시스템은 `POST /api/auth/signup` 요청을 처리하며 이메일 중복을 감지한다(BR-01).
3. 시스템은 409 Conflict 응답을 반환한다.
4. 화면에 "이미 사용 중인 이메일입니다." 오류 메시지가 표시되고 폼은 초기화되지 않은 채 유지된다.

**예외 흐름 (Alternative Flow)**
- 없음.

**사후 조건 (Postcondition)**
- DB에 중복 레코드가 생성되지 않는다.
- 김민영은 로그인 페이지 링크를 통해 기존 계정으로 로그인하거나, 다른 이메일로 재가입을 시도할 수 있다.

### SCN-03: 로그인 및 할일 목록 진입

- **관련 UC**: UC-02
- **관련 규칙**: BR-02, NFR-03, NFR-01

**사전 조건 (Precondition)**
- 김민영은 SCN-01을 통해 계정을 보유하고 있다.
- 현재 비인증 상태이며 `/login` 페이지에 접속해 있다.

**트리거 (Trigger)**
- 아침 출근 후 업무 시작 전 오늘 처리할 할일을 확인하기 위해 김민영이 PC 브라우저에서 TodoListApp에 접속한다.

**정상 흐름 (Main Flow)**
1. `/login` 페이지에서 이메일(`minkim@example.com`)과 비밀번호(`Planning123`)를 입력하고 "로그인" 버튼을 클릭한다.
2. 시스템은 `POST /api/auth/login` 요청을 처리한다. bcrypt로 비밀번호를 검증한 후 Access Token과 Refresh Token을 발급한다.
3. Access Token은 클라이언트 메모리(Zustand)에 저장되고, Refresh Token은 HttpOnly + Secure + SameSite=Strict 쿠키로 설정된다. 서버 DB의 `refresh_tokens` 테이블에도 SHA-256 해시값이 저장된다.
4. 로그인 성공 후 브라우저는 자동으로 `/` (할일 목록 페이지)로 이동한다.
5. 할일 목록 페이지는 `GET /api/todos` 요청으로 김민영의 할일 목록을 가져와 표시한다. 응답 시간은 500ms 이하로 유지된다(NFR-01).
6. 목록에 등록된 할일이 없다면 "아직 등록된 할일이 없습니다." 안내 문구가 표시된다.

**예외 흐름 (Alternative Flow)**
- **A1. 비밀번호 오류**: 잘못된 비밀번호 입력 시 시스템은 401 응답과 함께 "이메일 또는 비밀번호가 올바르지 않습니다." 메시지를 표시한다.
- **A2. 미가입 이메일**: 존재하지 않는 이메일 입력 시 보안상 동일한 오류 메시지를 표시하여 이메일 존재 여부를 노출하지 않는다.
- **A3. Rate Limiting 초과**: 분당 10회 이상 로그인 시도 시 429 응답과 재시도 대기 안내가 표시된다(NFR-03).
- **A4. 비인증 상태에서 보호 페이지 직접 접근**: 비인증 상태에서 `/` 에 직접 URL 접근 시 시스템은 `/login`으로 리다이렉트한다(BR-02, NFR-06).

**사후 조건 (Postcondition)**
- 김민영은 인증된 세션을 보유하고 할일 목록 페이지를 사용할 수 있는 상태가 된다.
- Access Token은 메모리에, Refresh Token은 HttpOnly 쿠키에 저장된 상태다.

### SCN-04: 로그인 실패 및 재시도

- **관련 UC**: UC-02
- **관련 규칙**: NFR-03

**사전 조건 (Precondition)**
- 박준혁은 계정을 보유하고 있으나 비밀번호가 기억나지 않는 상황이다.

**트리거 (Trigger)**
- 오랜만에 서비스에 접속하려는 박준혁이 여러 비밀번호를 시도하다가 오류가 반복된다.

**정상 흐름 (Main Flow)**
1. `/login` 페이지에서 이메일과 추정 비밀번호를 입력하고 로그인을 시도한다.
2. 시스템이 401 응답을 반환하며 "이메일 또는 비밀번호가 올바르지 않습니다." 메시지가 표시된다.
3. 박준혁은 다른 비밀번호 조합으로 재시도한다.
4. 분당 10회 시도 초과 시 429 응답이 반환되고 "잠시 후 다시 시도해 주세요." 안내가 표시된다(NFR-03).
5. 대기 후 올바른 비밀번호로 재시도하여 로그인에 성공한다.

**예외 흐름 (Alternative Flow)**
- **A1. 비밀번호 분실**: 현재 Phase 1에서는 비밀번호 재설정 기능이 Out of Scope다. 비밀번호를 기억하지 못하는 경우 신규 가입 외 대안이 없다.

**사후 조건 (Postcondition)**
- 최종적으로 올바른 자격증명으로 로그인에 성공하면 인증 세션이 생성된다.
- Rate Limiting은 시도 횟수 초과 후 자동으로 카운터가 초기화된다.

### SCN-05: 개인정보(이름) 수정

- **관련 UC**: UC-03
- **관련 규칙**: NFR-06

**사전 조건 (Precondition)**
- 김민영은 로그인된 상태로 할일 목록 페이지를 사용 중이다.

**트리거 (Trigger)**
- 결혼 후 성이 바뀐 김민영이 앱에 등록된 자신의 이름을 "김민영"에서 "박민영"으로 변경하고자 한다.

**정상 흐름 (Main Flow)**
1. 상단 네비게이션의 프로필 메뉴를 클릭하여 `/profile` 페이지로 이동한다.
2. 현재 등록된 이름("김민영")이 이름 필드에 표시된다.
3. 이름 필드를 "박민영"으로 수정하고 "저장" 버튼을 클릭한다.
4. 시스템은 `PATCH /api/users/me` 요청으로 이름 필드를 업데이트한다.
5. 성공 응답(200)을 받으면 화면에 "개인정보가 수정되었습니다." 토스트 메시지가 표시된다.
6. 상단 네비게이션의 이름 표시도 즉시 "박민영"으로 갱신된다.

**예외 흐름 (Alternative Flow)**
- **A1. 이름 빈값 제출**: 이름 필드를 비운 채 저장 시 클라이언트 유효성 검사에서 오류 메시지를 표시하고 요청을 차단한다.
- **A2. 인증 만료 중 수정 시도**: Access Token이 만료된 경우 시스템은 자동으로 `POST /api/auth/refresh`를 통해 토큰을 재발급한 뒤 원래 요청을 재전송한다 (Token Rotation 적용).

**사후 조건 (Postcondition)**
- `users` 테이블의 `name` 컬럼이 "박민영"으로 갱신된다.
- `updated_at` 타임스탬프가 UTC 기준으로 업데이트된다(NFR-07).

### SCN-06: 비밀번호 변경

- **관련 UC**: UC-03
- **관련 규칙**: NFR-03, NFR-06

**사전 조건 (Precondition)**
- 김민영은 로그인된 상태로 `/profile` 페이지에 접근해 있다.

**트리거 (Trigger)**
- 보안 의식이 높은 김민영이 3개월마다 비밀번호를 바꾸는 습관에 따라 비밀번호를 변경하려 한다.

**정상 흐름 (Main Flow)**
1. `/profile` 페이지에서 "비밀번호 변경" 섹션을 클릭하여 폼을 펼친다.
2. 현재 비밀번호(`Planning123`), 새 비밀번호(`NewPlan456`), 새 비밀번호 확인(`NewPlan456`)을 순서대로 입력한다.
3. "변경" 버튼을 클릭한다.
4. 시스템은 `PATCH /api/users/me` 요청에서 현재 비밀번호를 bcrypt로 검증한다.
5. 검증 통과 후 새 비밀번호를 bcrypt(salt rounds 12 이상)로 해싱하여 저장한다(NFR-03).
6. "비밀번호가 변경되었습니다." 메시지가 표시된다.

**예외 흐름 (Alternative Flow)**
- **A1. 현재 비밀번호 불일치**: 현재 비밀번호를 잘못 입력하면 401 응답과 함께 "현재 비밀번호가 올바르지 않습니다." 오류 메시지가 표시된다.
- **A2. 새 비밀번호 형식 오류**: 새 비밀번호가 8자 미만이거나 영문+숫자 조합 조건을 충족하지 못하면 클라이언트 측에서 즉시 오류를 표시한다(NFR-03).
- **A3. 새 비밀번호 확인 불일치**: 새 비밀번호와 확인값이 다르면 폼 유효성 검사에서 "비밀번호가 일치하지 않습니다." 메시지를 표시한다.

**사후 조건 (Postcondition)**
- `users` 테이블의 `password_hash`가 새 비밀번호의 해시값으로 갱신된다.
- 기존 Refresh Token은 보안 정책에 따라 서버에서 무효화되며, 김민영은 재로그인하도록 안내받는다 (보안 강화 관점).

### SCN-07: 할일 등록 (날짜 포함)

- **관련 UC**: UC-04
- **관련 규칙**: BR-04, BR-05, NFR-07

**사전 조건 (Precondition)**
- 이서연은 로그인된 상태다. "업무", "개인" 카테고리가 이미 생성되어 있다.
- 스마트폰(Android Chrome)으로 서비스를 사용 중이다 (NFR-05, 360px 뷰포트).

**트리거 (Trigger)**
- 이동 중 지하철에서 이서연은 내일까지 제출해야 할 디자인 시안 작업을 잊을까봐 바로 할일을 등록하려 한다.

**정상 흐름 (Main Flow)**
1. 할일 목록 페이지(`/`) 하단의 "+" 버튼을 탭하여 `/todos/new` 페이지로 이동한다.
2. 제목 입력 필드에 "디자인 시안 v1 제출"을 입력한다.
3. 설명 필드에 "클라이언트 피드백 반영본 포함"을 선택적으로 입력한다.
4. 카테고리 드롭다운에서 "업무"를 선택한다. 목록에는 기본 카테고리와 본인 소유 카테고리만 표시된다(BR-04).
5. 시작일을 오늘 날짜로, 종료예정일을 내일 날짜로 설정한다.
6. "등록" 버튼을 탭한다.
7. 시스템은 `POST /api/todos` 요청으로 할일을 저장한다. 날짜는 UTC 기준으로 DB에 저장된다(NFR-07).
8. 성공 응답 후 할일 목록 페이지로 돌아가며, 새로 등록된 할일이 목록 상단에 표시된다.

**예외 흐름 (Alternative Flow)**
- **A1. 제목 미입력**: 제목 없이 "등록" 버튼 탭 시 "제목을 입력해 주세요." 유효성 오류가 표시된다.
- **A2. 카테고리 미선택**: 카테고리를 선택하지 않은 상태에서 제출 시 "카테고리를 선택해 주세요." 오류가 표시된다(BR-04).
- **A3. 날짜 범위 오류**: 종료예정일이 시작일보다 앞선 날짜로 설정될 경우 SCN-08 시나리오로 이어진다(BR-05).
- **A4. 네트워크 오류**: 지하철 터널 구간에서 네트워크 단절 발생 시 요청이 실패하고 "네트워크 오류가 발생했습니다. 다시 시도해 주세요." 안내가 표시된다.

**사후 조건 (Postcondition)**
- `todos` 테이블에 새 할일 레코드가 생성된다 (`is_completed = false`).
- `start_date`, `due_date`는 UTC 기준 DATE 타입으로 저장된다(NFR-07).
- Overdue 판단은 이서연의 기기 로컬 타임존(KST) 기준으로 클라이언트에서 수행된다(NFR-07).

### SCN-08: 잘못된 날짜 범위로 할일 등록 실패

- **관련 UC**: UC-04, UC-05
- **관련 규칙**: BR-05

**사전 조건 (Precondition)**
- 이서연은 로그인된 상태로 `/todos/new` 또는 `/todos/:id/edit` 페이지에 있다.

**트리거 (Trigger)**
- 급하게 입력하던 이서연이 실수로 시작일을 종료예정일보다 늦은 날짜로 설정했다.

**정상 흐름 (Main Flow)**
1. 시작일을 "2026-05-20", 종료예정일을 "2026-05-15"로 입력한다.
2. "등록" 또는 "저장" 버튼을 탭한다.
3. 시스템(클라이언트 또는 서버)이 날짜 규칙 위반을 감지한다(BR-05).
4. "종료예정일은 시작일과 같거나 이후 날짜여야 합니다." 오류 메시지가 종료예정일 필드 하단에 표시된다.
5. 폼은 입력값을 유지한 채 제출이 차단된다.
6. 이서연은 종료예정일을 "2026-05-20" 이후로 수정하여 정상 등록을 완료한다.

**예외 흐름 (Alternative Flow)**
- **A1. 클라이언트 우회 시도**: 클라이언트 검증을 우회하여 잘못된 날짜로 서버에 요청을 보낼 경우, 서버도 동일한 규칙으로 400 Bad Request를 반환한다.

**사후 조건 (Postcondition)**
- 유효하지 않은 날짜 범위의 할일은 DB에 저장되지 않는다.
- 수정 후 올바른 날짜 범위로 재시도하면 정상적으로 저장된다.

### SCN-09: 할일 수정

- **관련 UC**: UC-05
- **관련 규칙**: BR-03, BR-04, BR-05, NFR-06

**사전 조건 (Precondition)**
- 김민영은 로그인된 상태이며, 이전에 등록한 할일 "주간 보고서 작성"이 목록에 있다.

**트리거 (Trigger)**
- 회의 일정이 변경되어 할일의 종료예정일을 하루 연장해야 한다. 업무 중 PC 브라우저에서 빠르게 수정하려 한다.

**정상 흐름 (Main Flow)**
1. 할일 목록(`/`)에서 "주간 보고서 작성" 항목의 편집 아이콘을 클릭한다.
2. `/todos/:id/edit` 페이지로 이동하며 현재 저장된 값들이 폼에 자동 입력된다.
3. 종료예정일 필드를 기존 날짜에서 하루 뒤 날짜로 변경한다.
4. 필요에 따라 카테고리를 "업무"에서 "개인"으로 변경한다(BR-04, 본인 소유 카테고리).
5. "저장" 버튼을 클릭한다.
6. 시스템은 `PATCH /api/todos/:id` 요청으로 변경사항을 저장한다. 서버는 해당 할일의 `user_id`가 현재 로그인 사용자와 일치하는지 검증한다(BR-03).
7. 성공 응답 후 목록 페이지로 돌아가며 수정된 내용이 반영된다.

**예외 흐름 (Alternative Flow)**
- **A1. 타 사용자의 할일 수정 시도**: 다른 사용자의 할일 ID로 수정 요청을 보낼 경우 서버는 403 Forbidden을 반환하고 "접근 권한이 없습니다." 메시지가 표시된다(BR-03, NFR-06).
- **A2. 존재하지 않는 할일**: 이미 삭제된 할일 ID로 요청 시 404 Not Found가 반환된다.

**사후 조건 (Postcondition)**
- `todos` 테이블의 해당 레코드가 변경된 값으로 업데이트된다.
- `updated_at` 타임스탬프가 UTC 기준으로 갱신된다(NFR-07).

### SCN-10: 할일 완료 처리 및 토글

- **관련 UC**: UC-06
- **관련 규칙**: BR-03

**사전 조건 (Precondition)**
- 박준혁은 로그인된 상태로 할일 목록 페이지(`/`)를 보고 있다. "팀 주간 회의 자료 준비" 할일이 미완료 상태로 존재한다.

**트리거 (Trigger)**
- 회의를 마친 박준혁이 점심시간에 스마트폰으로 완료된 할일을 체크하려 한다.

**정상 흐름 (Main Flow)**
1. 할일 목록에서 "팀 주간 회의 자료 준비" 항목 왼쪽의 체크박스를 탭한다.
2. 시스템은 `PATCH /api/todos/:id` 요청으로 `is_completed` 값을 `true`로 변경한다.
3. 체크박스가 체크 표시로 바뀌고 할일 제목에 취소선이 생긴다. 완료 처리된 할일은 목록 하단으로 이동할 수 있다.
4. 실수로 완료 처리했다고 판단한 박준혁이 동일한 체크박스를 다시 탭한다.
5. 시스템은 `PATCH /api/todos/:id` 요청으로 `is_completed` 값을 다시 `false`로 변경한다 (완료 ↔ 미완료 토글).
6. 체크박스가 빈 상태로 돌아가고 취소선이 사라진다.

**예외 흐름 (Alternative Flow)**
- **A1. 타 사용자의 할일**: 서버는 요청한 할일의 소유권을 검증하며, 본인 소유가 아닌 경우 403을 반환한다(BR-03).
- **A2. 빠른 연속 탭**: 짧은 시간 내 연속 탭으로 상태 불일치가 발생할 수 있으므로, 클라이언트는 낙관적 업데이트(Optimistic Update) 후 서버 응답으로 최종 상태를 동기화한다.

**사후 조건 (Postcondition)**
- `todos` 테이블의 `is_completed` 값이 최종 탭 결과에 따라 `true` 또는 `false`로 저장된다.
- `updated_at`이 UTC 기준으로 갱신된다(NFR-07).

### SCN-11: 할일 삭제

- **관련 UC**: UC-07
- **관련 규칙**: BR-03, NFR-06

**사전 조건 (Precondition)**
- 박준혁은 로그인된 상태로 할일 목록 페이지(`/`)에 있다. 더 이상 필요하지 않은 할일 "구형 보고서 양식 검토"가 목록에 존재한다.

**트리거 (Trigger)**

**정상 흐름 (Main Flow)**
1. 할일 목록에서 "구형 보고서 양식 검토" 항목의 삭제 아이콘(휴지통)을 클릭한다.
2. 확인 다이얼로그가 표시된다: "이 할일을 삭제하시겠습니까? 이 작업은 되돌릴 수 없습니다."

5. 성공 응답(200 또는 204)을 받으면 해당 항목이 목록에서 즉시 사라진다.

**예외 흐름 (Alternative Flow)**
- **A1. 삭제 취소**: 확인 다이얼로그에서 "취소" 버튼을 클릭하면 아무 변경 없이 목록 화면으로 돌아간다.

- **A3. 이미 삭제된 할일**: 동일 할일에 대해 중복 삭제 요청 시 404를 반환하며 클라이언트는 이미 목록에서 제거된 상태를 유지한다.

**사후 조건 (Postcondition)**
- `todos` 테이블에서 해당 레코드가 영구 삭제된다.
- 할일 목록 페이지가 갱신되어 삭제된 항목이 더 이상 표시되지 않는다.

### SCN-12: 할일 목록 필터링 조회

- **관련 규칙**: BR-08, NFR-01, NFR-07

- 화면에는 선택한 필터 조건에 부합하는 할일만 표시된다.
- 필터 상태는 URL 쿼리 파라미터에 반영되어 공유 또는 새로고침 시에도 동일한 필터가 유지될 수 있다.

### SCN-13: 카테고리 추가

- **관련 규칙**: BR-03, BR-06, NFR-06

**사후 조건 (Postcondition)**
- `categories` 테이블에 `user_id = 이서연의 ID`인 새 카테고리 레코드가 생성된다.
- 이후 할일 등록/수정 시 카테고리 선택 드롭다운에 "사이드프로젝트"가 표시된다.

### SCN-14: 카테고리 중복명 등록 실패

- **관련 UC**: UC-09
- **관련 규칙**: BR-03

**사전 조건 (Precondition)**
- 이서연은 이미 "업무" 카테고리를 보유하고 있다.

**트리거 (Trigger)**
- 이서연이 실수로 "업무" 카테고리를 중복으로 추가하려 한다.

**정상 흐름 (Main Flow)**
1. `/categories` 페이지에서 카테고리명 필드에 "업무"를 입력하고 "추가" 버튼을 탭한다.
2. 시스템은 `POST /api/categories` 요청을 처리하며 동일 사용자의 기존 카테고리명과 비교한다.
3. 중복이 감지되어 409 Conflict 응답을 반환한다.
4. "이미 존재하는 카테고리명입니다." 오류 메시지가 표시된다.

**예외 흐름 (Alternative Flow)**
- **A1. 대소문자 처리**: "업무"와 "업무"를 별개로 처리할지는 서버 측 비교 로직에 따른다. PRD 기준 동일 사용자 내 중복 불가 원칙이 우선한다.

**사후 조건 (Postcondition)**
- 중복 카테고리 레코드는 생성되지 않는다.
- 이서연은 다른 카테고리명으로 재시도할 수 있다.

### SCN-15: 카테고리 삭제 및 할일 자동 재배정

- **관련 UC**: UC-10
- **관련 규칙**: BR-06, BR-07, NFR-04

**사전 조건 (Precondition)**
- 박준혁은 로그인된 상태이며, "프로젝트A" 카테고리 아래 "킥오프 미팅 준비", "요구사항 정리" 할일이 등록되어 있다. 해당 프로젝트가 종료되었다.

**트리거 (Trigger)**
- 프로젝트A가 완전히 종료되어 박준혁이 관련 카테고리를 정리하려 한다. 할일 자체는 기록 보존 목적으로 삭제하지 않는다.

**정상 흐름 (Main Flow)**
1. `/categories` 페이지에서 "프로젝트A" 카테고리 옆의 삭제 버튼을 클릭한다.
2. 확인 다이얼로그가 표시된다: "'프로젝트A' 카테고리를 삭제하면, 해당 카테고리의 할일 2개가 '기본' 카테고리로 이동됩니다. 계속하시겠습니까?"
3. 박준혁은 "삭제" 버튼을 클릭하여 확정한다.
4. 시스템은 `DELETE /api/categories/:id` 요청을 처리한다. 단일 DB 트랜잭션 내에서 다음을 수행한다(BR-07, NFR-04):
   a. "프로젝트A" 카테고리에 속한 모든 할일의 `category_id`를 기본 카테고리 ID로 업데이트한다.
   b. "프로젝트A" 카테고리 레코드를 삭제한다.
5. 두 작업 모두 성공 시 커밋하고, 하나라도 실패 시 전체 롤백한다(NFR-04).
6. 성공 응답 후 카테고리 목록에서 "프로젝트A"가 사라진다.
7. 할일 목록에서 기존 "프로젝트A" 소속 할일들은 "기본" 카테고리 소속으로 표시된다.

**예외 흐름 (Alternative Flow)**
- **A1. 삭제 취소**: 확인 다이얼로그에서 "취소" 클릭 시 아무 변경 없이 원래 상태로 돌아간다.
- **A2. 트랜잭션 실패**: DB 오류로 트랜잭션이 실패할 경우 전체 롤백 후 "카테고리 삭제 중 오류가 발생했습니다. 다시 시도해 주세요." 메시지가 표시된다(NFR-04). 카테고리와 할일 모두 원상 유지된다.

**사후 조건 (Postcondition)**
- `categories` 테이블에서 "프로젝트A" 레코드가 삭제된다.
- 기존 "프로젝트A" 소속 할일들의 `category_id`가 기본 카테고리 ID로 업데이트된다(BR-07).
- 할일 데이터는 손실 없이 보존된다.

### SCN-16: 기본 카테고리 삭제 시도 실패

- **관련 UC**: UC-10
- **관련 규칙**: BR-06

**사전 조건 (Precondition)**
- 박준혁은 로그인된 상태이며, `/categories` 페이지를 보고 있다.

**트리거 (Trigger)**
- 박준혁이 카테고리 목록을 정리하면서 "기본" 카테고리도 삭제해 보려 한다.

**정상 흐름 (Main Flow)**
1. "기본" 카테고리 항목에서 삭제 버튼이 비활성화(disabled) 상태로 표시된다(BR-06).
2. 비활성화된 버튼에 마우스를 올리면 "기본 카테고리는 삭제할 수 없습니다." 툴팁이 표시된다.
3. 클릭이 되지 않아 삭제 요청 자체가 발생하지 않는다.

**예외 흐름 (Alternative Flow)**
- **A1. API 직접 호출**: 클라이언트 우회를 통해 기본 카테고리 삭제를 API로 직접 요청할 경우, 서버는 `is_default = true` 여부를 검사하여 403 Forbidden을 반환한다(BR-06).

**사후 조건 (Postcondition)**
- 기본 카테고리는 삭제되지 않고 유지된다.
- 시스템 내 할일 재배정 대상으로서의 기본 카테고리 기능이 보존된다.

### SCN-17: 로그아웃

- **관련 UC**: UC-11
- **관련 규칙**: NFR-03 (토큰 보안)

**사전 조건 (Precondition)**
- 김민영은 회사 공용 PC에서 로그인된 상태로 TodoListApp을 사용 중이다.

**트리거 (Trigger)**
- 퇴근 시간이 되어 공용 PC에서 작업을 마친 김민영이 다른 사람이 자신의 계정에 접근하지 못하도록 로그아웃하려 한다.

**정상 흐름 (Main Flow)**
1. 상단 네비게이션의 프로필 메뉴에서 "로그아웃"을 클릭한다.
2. 시스템은 `POST /api/auth/logout` 요청을 전송한다.
3. 서버는 DB의 `refresh_tokens` 테이블에서 해당 사용자의 현재 Refresh Token 레코드를 즉시 삭제하여 무효화한다.
4. 클라이언트는 메모리(Zustand)에 저장된 Access Token을 초기화한다.
5. HttpOnly 쿠키의 Refresh Token이 만료(삭제)된다.
6. 브라우저는 `/login` 페이지로 리다이렉트된다.
7. 로그아웃 이후 브라우저 뒤로가기로 이전 페이지에 접근하면 인증이 없으므로 다시 `/login`으로 리다이렉트된다(NFR-06).

**예외 흐름 (Alternative Flow)**
- **A1. 네트워크 오류 시 로그아웃**: 서버 요청이 실패하더라도 클라이언트 측 토큰(Access Token 메모리, 쿠키)은 삭제하여 로컬에서는 로그아웃 처리한다. 서버의 Refresh Token은 만료 시점에 자동으로 정리된다.

**사후 조건 (Postcondition)**
- 서버 DB의 Refresh Token 레코드가 삭제된다.
- 클라이언트 메모리의 Access Token이 초기화된다.
- 김민영은 비인증 상태가 되며 보호 페이지에 접근할 수 없다.

### SCN-18: 회원 탈퇴

- **관련 UC**: UC-12
- **관련 규칙**: NFR-04, NFR-03

**사전 조건 (Precondition)**
- 이서연은 로그인된 상태이며, 앱을 더 이상 사용하지 않기로 결정했다. 할일 3개, 카테고리 2개, 활성 Refresh Token 1개가 계정에 연결되어 있다.

**트리거 (Trigger)**
- 개인 데이터가 서버에 남아있는 것이 불편한 이서연이 계정과 모든 데이터를 완전히 삭제하려 한다.

**정상 흐름 (Main Flow)**
1. `/profile` 페이지 하단의 "회원 탈퇴" 섹션을 클릭하여 펼친다.
2. 본인 확인을 위해 현재 비밀번호를 입력하는 필드와 안내 문구가 표시된다: "탈퇴 시 모든 데이터(할일, 카테고리, 계정 정보)가 영구 삭제됩니다."
3. 이서연은 현재 비밀번호를 입력하고 "탈퇴 확인" 버튼을 클릭한다.
4. 최종 확인 다이얼로그가 표시된다: "정말로 탈퇴하시겠습니까? 이 작업은 되돌릴 수 없습니다."
5. "탈퇴" 버튼을 클릭한다.
6. 시스템은 `DELETE /api/users/me` 요청으로 비밀번호를 검증한다(NFR-03).
7. 단일 DB 트랜잭션 내에서 다음 순서로 원자적 삭제를 수행한다(NFR-04):
   a. `refresh_tokens` 테이블에서 이서연의 모든 Refresh Token 삭제
   b. `todos` 테이블에서 이서연의 모든 할일 삭제
   c. `categories` 테이블에서 이서연 소유의 모든 카테고리 삭제
   d. `users` 테이블에서 이서연의 계정 레코드 삭제
8. 모든 단계 성공 시 트랜잭션 커밋. 하나라도 실패 시 전체 롤백(NFR-04).
9. 성공 응답 후 클라이언트의 모든 인증 토큰을 초기화하고 `/login` 페이지로 리다이렉트한다.
10. "탈퇴가 완료되었습니다. 그동안 이용해 주셔서 감사합니다." 안내 메시지가 표시된다.

**예외 흐름 (Alternative Flow)**
- **A1. 비밀번호 불일치**: 입력한 비밀번호가 올바르지 않으면 401 응답과 함께 "비밀번호가 올바르지 않습니다." 오류가 표시되고 탈퇴 프로세스가 중단된다.
- **A2. 탈퇴 취소**: 최종 확인 다이얼로그에서 "취소"를 클릭하면 아무 데이터도 삭제되지 않고 프로필 페이지로 돌아간다.
- **A3. 트랜잭션 실패**: DB 오류 발생 시 모든 삭제 작업이 롤백되어 계정과 데이터가 원상 보존된다. "탈퇴 처리 중 오류가 발생했습니다. 다시 시도해 주세요." 메시지가 표시된다(NFR-04).

**사후 조건 (Postcondition)**
- `users`, `todos`, `categories`, `refresh_tokens` 테이블에서 이서연의 모든 데이터가 단일 트랜잭션으로 완전 삭제된다(NFR-04).
- 이서연은 비인증 상태가 되며, 이전 이메일로 재가입이 가능하다(BR-01 — 이메일 고유성은 계정 삭제 후 해제됨).
- 클라이언트 메모리와 쿠키의 모든 인증 정보가 초기화된다.
