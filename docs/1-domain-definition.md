# TodoListApp 도메인 정의서

> 버전: 0.1.0-draft | 작성일: 2026-05-12 | 작성자: Business Analyst

---

## 1. 프로젝트 개요

| 항목 | 내용 |
|------|------|
| 프로젝트명 | TodoListApp |
| 목적 | 인증된 사용자가 개인 할일을 카테고리 단위로 체계적으로 관리할 수 있는 웹 애플리케이션 |
| 대상 사용자 | 개인 할일·일정 관리가 필요한 일반 사용자 |
| 핵심 가치 | 개인화된 할일 데이터 보호 + 카테고리 기반 유연한 분류 |

---

## 2. 문제 정의

| # | 문제 | 영향 |
|---|------|------|
| P-01 | 개인별로 할일을 독립적으로 관리할 수 있는 적합한 앱 부재 | 타인과 데이터가 혼재되어 프라이버시 및 가시성 저하 |
| P-02 | 인증된 사용자 단위로 데이터베이스에 할일을 저장·격리하는 앱 부재 | 사용자별 데이터 무결성과 보안 보장 불가 |
| P-03 | 카테고리 별로 할일을 등록·조회할 수 있는 앱 부재 | 할일이 단순 목록으로만 관리되어 맥락적 분류 불가 |

## 3. 핵심 도메인 개념

### 3.1 도메인 용어 사전 (Ubiquitous Language)

| 용어 | 영문 | 정의 |
|------|------|------|
| 사용자 | User | 회원가입 후 인증을 통해 앱을 사용하는 주체. 각 사용자의 할일 데이터는 독립적으로 격리됨 |
| 인증 | Authentication | 사용자가 이메일·비밀번호로 자신의 신원을 확인하는 절차 |
| 할일 | Todo | 사용자가 수행해야 할 개별 작업 단위. 제목·설명·기간·카테고리·완료 여부를 포함 |
| 카테고리 | Category | 할일을 의미 단위로 묶는 분류 레이블. 기본 카테고리와 사용자 정의 카테고리로 구분 |
| 기본 카테고리 | Default Category | 시스템이 모든 신규 사용자에게 자동으로 제공하는 사전 정의 카테고리 |
| 사용자 정의 카테고리 | Custom Category | 인증된 사용자가 직접 생성한 카테고리 |
| 시작일 | Start Date | 할일을 시작하는 날짜 |
| 종료예정일 | Due Date | 할일을 완료해야 하는 목표 날짜 |
| 완료 여부 | Completion Status | 할일의 처리 상태 (미완료 / 완료) |
| 기간 만료 | Overdue | 현재 날짜가 종료예정일을 초과하고 완료되지 않은 상태 |
| 개인정보 | Profile | 사용자의 이름, 이메일 등 계정 식별 정보 |

## 4. 도메인 모델

### 4.1 주요 엔티티

```
User (사용자)
├── id              : 식별자 (PK)
├── email           : 이메일 (고유)
├── password_hash   : 암호화된 비밀번호
├── name            : 이름
└── created_at      : 가입일시

Category (카테고리)
├── id              : 식별자 (PK)
├── user_id         : 소유 사용자 (FK → User) — NULL이면 기본 카테고리
├── name            : 카테고리명
├── is_default      : 기본 카테고리 여부
└── created_at      : 생성일시

Todo (할일)
├── id              : 식별자 (PK)
├── user_id         : 소유 사용자 (FK → User)
├── category_id     : 카테고리 (FK → Category)
├── title           : 할일 제목
├── description     : 설명
├── start_date      : 시작일
├── due_date        : 종료예정일
├── is_completed    : 완료 여부 (boolean)
└── created_at      : 등록일시
```

### 4.2 엔티티 관계

```
User ──(1:N)──> Todo
User ──(1:N)──> Category (사용자 정의)
Category ──(1:N)──> Todo
System ──(1:N)──> Category (기본 카테고리, user_id = NULL)
```

### 4.3 주요 제약

- 할일은 반드시 하나의 카테고리에 속해야 한다.
- 카테고리는 해당 사용자가 소유한 카테고리이거나 기본 카테고리이어야 한다.
- 사용자는 자신의 할일과 카테고리에만 접근할 수 있다.

## 5. 핵심 비즈니스 규칙

| ID | 규칙 | 범주 |
|----|------|------|
| BR-01 | 이메일 주소는 시스템 전체에서 고유해야 한다 | 사용자 |
| BR-02 | 인증되지 않은 사용자는 할일 및 카테고리에 접근할 수 없다 | 인증 |
| BR-03 | 사용자는 자신이 소유한 할일과 카테고리만 조회·수정·삭제할 수 있다 | 접근 제어 |
| BR-04 | 할일 등록 시 카테고리는 필수이며, 기본 카테고리 또는 본인 소유 카테고리만 선택 가능하다 | 할일 |
| BR-05 | 종료예정일은 시작일과 같거나 이후 날짜여야 한다 | 할일 |
| BR-06 | 기본 카테고리는 사용자가 수정하거나 삭제할 수 없다 | 카테고리 |
| BR-07 | 카테고리가 삭제될 경우, 해당 카테고리에 속한 할일은 기본 카테고리로 자동 재배정된다 | 카테고리 |
| BR-08 | 할일 목록은 카테고리 / 기간 만료 여부 / 완료 여부 기준으로 필터링할 수 있다 | 조회 |

## 6. 주요 유스케이스

| ID | 유스케이스 | 행위자 | 핵심 흐름 |
|----|-----------|--------|-----------|
| UC-01 | 회원가입 | 비인증 사용자 | 이메일·비밀번호·이름 입력 → 중복 이메일 검증 → 계정 생성 |
| UC-02 | 로그인 | 비인증 사용자 | 이메일·비밀번호 입력 → 인증 토큰 발급 |
| UC-03 | 개인정보 수정 | 인증된 사용자 | 이름·비밀번호 변경 → 저장 |
| UC-04 | 할일 등록 | 인증된 사용자 | 제목·설명·시작일·종료예정일·카테고리 입력 → 저장 |
| UC-05 | 할일 수정 | 인증된 사용자 | 기존 할일 항목 선택 → 필드 수정 → 저장 |
| UC-06 | 할일 완료 처리 | 인증된 사용자 | 할일 선택 → 완료 상태로 변경 |
| UC-07 | 할일 삭제 | 인증된 사용자 | 할일 선택 → 삭제 확인 → 제거 |
| UC-08 | 할일 목록 조회 | 인증된 사용자 | 카테고리·기간 만료 여부·완료 여부 필터 적용 → 목록 출력 |
| UC-09 | 카테고리 추가 | 인증된 사용자 | 카테고리명 입력 → 중복 검증 → 저장 |
| UC-10 | 카테고리 삭제 | 인증된 사용자 | 사용자 정의 카테고리 선택 → 삭제 → 소속 할일 기본 카테고리 재배정 |

## 7. 도메인 경계 (Bounded Context)

```
┌─────────────────────────────────────────────────────────────┐
│                        TodoListApp                          │
│                                                             │
│   ┌──────────────────┐       ┌───────────────────────────┐  │
│   │  Identity Context │       │    Todo Management Context │  │
│   │  (신원·인증)       │       │    (할일·카테고리 관리)     │  │
│   │                  │       │                           │  │
│   │  - 회원가입       │──────>│  - 할일 CRUD              │  │
│   │  - 로그인·로그아웃 │ 인증  │  - 카테고리 관리           │  │
│   │  - 개인정보 수정   │  토큰 │  - 할일 목록 필터링        │  │
│   └──────────────────┘       └───────────────────────────┘  │
└─────────────────────────────────────────────────────────────┘
```

| Bounded Context | 책임 | 핵심 엔티티 |
|-----------------|------|------------|
| Identity Context | 사용자 등록, 인증, 개인정보 관리 | User |
| Todo Management Context | 할일 및 카테고리의 생성·조회·수정·삭제, 필터링 | Todo, Category |

두 컨텍스트는 JWT(Access Token + Refresh Token)를 경계로 통합되며, Todo Management Context는 Identity Context가 발급한 JWT를 신뢰하여 사용자 식별에 활용한다.
