## 1. 완전성 (Completeness) — 72점

### 잘된 점

- 프로젝트 개요, 문제 정의, 도메인 용어, 도메인 모델, 비즈니스 규칙, 유스케이스, Bounded Context까지 7개 섹션을 포함하여 기본 골격이 충실하다.
- 핵심 엔티티(User, Category, Todo)의 주요 속성이 명시되어 있으며, 관계(1:N)도 도식화되어 있다.
- 용어 사전(Ubiquitous Language)이 별도 섹션으로 분리되어 있어 팀 공용 언어 정립 의도가 명확하다.
- 비즈니스 규칙(BR-01~BR-08)이 ID 체계로 목록화되어 있다.

### 개선이 필요한 점

- **비기능 요구사항(NFR) 전면 누락**: 성능(응답 시간, 동시 사용자 수), 보안(비밀번호 해시 알고리즘, 토큰 만료 정책), 가용성, 확장성에 대한 기준이 전혀 없다.
- **데이터 유효성 제약 미흡**: 이메일 형식, 비밀번호 복잡도 규칙, 카테고리명 최대 길이, 할일 제목 최대 길이 등 입력값 제약이 정의되어 있지 않다.
- **예외 및 오류 시나리오 부재**: 각 유스케이스의 실패 흐름(예: 중복 이메일, 잘못된 비밀번호, 존재하지 않는 카테고리 선택)이 기술되어 있지 않다.
- **로그아웃 유스케이스 누락**: Bounded Context 도식에 "로그인·로그아웃"이 언급되었으나, UC 목록에 로그아웃(UC)이 없다.
- **감사(Audit) 및 이력 관리 누락**: Todo 수정 이력이나 삭제 이력에 대한 도메인 요구사항이 없다.
- **외부 시스템 연동 범위 미정의**: 이메일 인증, 소셜 로그인, 알림(Push/Email) 등 외부 연동 포함 여부가 명시되지 않았다.

### 구체적인 개선 제안

1. 섹션 8로 "비기능 요구사항(NFR)" 섹션을 추가하고, 응답 시간 목표(예: API 95th percentile < 500ms), 동시 접속자 목표, 데이터 보존 정책을 기술하라.
2. 각 엔티티 속성에 유효성 제약(길이, 형식, 필수 여부)을 컬럼으로 추가하라.
3. 유스케이스 표에 "예외 흐름" 컬럼을 추가하거나, 주요 UC에 대해 별도 상세 명세를 작성하라.
4. UC-02 로그인 행 옆에 UC-02b 로그아웃을 추가하고, Identity Context 책임에 명시하라.
5. 범위 결정 섹션(In-Scope / Out-of-Scope)을 추가하여 외부 연동 여부를 명확히 선언하라.

## 2. 명확성 (Clarity) — 81점

### 잘된 점

- 도메인 용어 사전이 한국어-영문 대응으로 작성되어 언어적 혼란을 최소화한다.
- 엔티티 관계가 텍스트 다이어그램으로 시각화되어 있어 읽기 쉽다.
- 비즈니스 규칙이 ID(BR-xx) 기반으로 구분되어 있고 범주 컬럼이 있어 검색·분류가 용이하다.
- Bounded Context 경계가 박스 다이어그램으로 표현되어 컨텍스트 간 책임 분리가 직관적이다.

### 개선이 필요한 점

- **BR-07의 "기본 카테고리" 특정 필요**: "기본 카테고리로 자동 재배정"된다고 명시했으나, 어떤 기본 카테고리(예: "미분류")로 재배정되는지 대상이 불명확하다.
- **`is_default` 와 `user_id = NULL` 중복 설계**: 기본 카테고리를 판단하는 기준이 두 가지 속성으로 분산되어 있어 구현 시 혼선이 발생할 수 있다. 하나의 기준으로 단일화해야 한다.
- **인증 토큰 방식 미결정**: "JWT 또는 세션"으로 기술되어 있는데, 도메인 정의 단계에서도 방향을 결정하거나 명시적으로 "추후 결정" 표시가 필요하다.
- **완료 처리의 역방향(완료 취소) 미정의**: UC-06은 완료 처리만 있고, 완료 상태를 미완료로 되돌리는 행위가 허용되는지 불명확하다.
- **카테고리 수정 유스케이스 누락**: 카테고리 추가(UC-09)와 삭제(UC-10)는 있지만 이름 변경 등 수정 유스케이스가 없다. 허용 여부 자체를 명시해야 한다.

### 구체적인 개선 제안

1. BR-07을 "해당 카테고리에 속한 할일은 시스템 기본 카테고리 '미분류(Uncategorized)'로 자동 재배정된다"와 같이 대상을 명시하라.
2. 기본 카테고리 판별 기준을 `is_default = true`로 단일화하고, `user_id = NULL` 설계 방식을 폐기하거나 주석으로 설계 결정 이유를 기술하라.
3. Bounded Context 섹션에 "인증 토큰 방식: TBD (JWT vs 세션 쿠키)" 항목을 추가하고, 의사결정 예정일을 기록하라.
4. BR-06 또는 신규 BR로 "사용자 정의 카테고리는 이름 변경이 가능하다"와 "완료된 할일은 미완료 상태로 되돌릴 수 있다/없다"를 명시하라.

## 3. 추적성 (Traceability) — 63점

### 잘된 점

- 문제(P-01~P-03), 비즈니스 규칙(BR-01~BR-08), 유스케이스(UC-01~UC-10)에 고유 ID가 부여되어 있어 추적의 기반 구조가 존재한다.
- 유스케이스와 행위자(Actor)가 매핑되어 있어 WHO-WHAT 연결이 가능하다.

### 개선이 필요한 점

- **문제-유스케이스 매핑 부재**: P-01, P-02, P-03이 각각 어떤 UC를 통해 해결되는지 연결 관계가 없다.
- **BR-UC 연결 부재**: 각 비즈니스 규칙이 어떤 유스케이스에서 적용되는지 매핑 테이블이 없다.
- **엔티티-UC 연결 부재**: 유스케이스별로 어떤 엔티티를 CRUD하는지 명시되어 있지 않다.
- **요구사항 ID 체계 미통합**: P, BR, UC가 각각 독립적인 ID를 가지며, 이를 연결하는 상위 요구사항 ID(REQ-xx) 체계가 없다.
- **구현(기술 스펙)으로의 추적 링크 없음**: API 엔드포인트, DB 테이블, 화면 ID 등 하위 산출물로의 연결 고리가 전혀 없다.

### 구체적인 개선 제안

1. 문제 정의 섹션 표에 "해결 UC" 컬럼을 추가하라. (예: P-01 → UC-01, UC-02 / P-03 → UC-08, UC-09)
2. 비즈니스 규칙 표에 "적용 UC" 컬럼을 추가하라. (예: BR-05 → UC-04, UC-05)
3. 유스케이스 표에 "관련 엔티티" 및 "적용 BR" 컬럼을 추가하여 양방향 추적이 가능하도록 하라.
4. 상위 수준 요구사항 ID(REQ-xx) 체계를 도입하고, 문서 버전이 올라갈 때 하위 기술 문서(API 명세, ERD)와의 링크를 추가하라.

## 4. 검증 가능성 (Verifiability) — 55점

### 잘된 점

- BR-05("종료예정일은 시작일과 같거나 이후 날짜여야 한다")와 BR-01("이메일 주소는 시스템 전체에서 고유해야 한다")은 명확한 조건으로 기술되어 테스트 케이스 도출이 가능하다.
- BR-07("소속 할일 기본 카테고리 재배정")은 결과 상태가 명시되어 있어 검증 조건 도출이 상대적으로 용이하다.

### 개선이 필요한 점

- **수용 기준(Acceptance Criteria) 전면 부재**: 어떤 유스케이스도 "완료 판단 기준"이 기술되어 있지 않다. QA팀이 테스트를 설계하려면 별도 해석이 필요하다.
- **BR의 모호한 표현**: BR-02("접근할 수 없다")는 HTTP 401 응답인지, 빈 목록 반환인지, 리다이렉트인지 정의되지 않아 테스트 기준이 불명확하다.
- **성공/실패 시나리오 미정의**: 회원가입 성공 시 반환값, 실패 시 오류 코드 등 검증 가능한 출력 기준이 없다.
- **필터링 결과 검증 기준 부재**: BR-08에서 필터링 기준을 나열했지만, 복합 필터(AND/OR 조합) 동작 방식이 미정의이다.
- **성능·보안 관련 테스트 기준 없음**: 비기능 요구사항이 없으므로 당연히 관련 검증 기준도 없다.

### 구체적인 개선 제안

1. 각 UC에 "수용 기준(AC)" 서브섹션을 추가하라. 예시:
   - UC-01 수용 기준: (AC-01-01) 이미 사용 중인 이메일로 가입 시도 시 HTTP 409 및 오류 메시지 반환 / (AC-01-02) 유효한 정보 입력 시 계정 생성 후 HTTP 201 반환
2. BR-02를 "인증되지 않은 요청은 HTTP 401 Unauthorized를 반환한다"로 구체화하라.
3. BR-08에 복합 필터 동작 정책을 추가하라. (예: "필터 항목 간 AND 조건으로 동작한다")
4. 비기능 요구사항 섹션 추가 후, 성능 목표 수치를 명시하여 부하 테스트 기준으로 활용 가능하게 하라.

## 5. 유지보수성 (Maintainability) — 70점

### 잘된 점

- 문서 헤더에 버전(0.1.0-draft), 작성일, 작성자가 명시되어 있어 기본적인 메타데이터가 있다.
- 각 항목에 고유 ID(BR-xx, UC-xx, P-xx)가 부여되어 변경 시 영향 범위 식별이 가능하다.
- 섹션 구조가 논리적으로 분리되어 있어 특정 섹션만 독립적으로 업데이트하기 용이하다.
- 문서 말미에 "초안(draft)이며 확정 전 검토를 거친다"는 상태 표시가 있다.

### 개선이 필요한 점

- **변경 이력(Change Log) 부재**: 어떤 날짜에, 누가, 무엇을 변경했는지 추적할 수 없다.
- **이해관계자 검토·승인 기록 없음**: 누가 이 문서를 검토하고 승인했는지 기록이 없어 책임 소재가 불명확하다.
- **관련 문서 참조 없음**: ERD, API 명세, 화면 설계서, 요구사항 명세서 등 연관 문서와의 링크가 없다.
- **용어 확장 구조 미비**: 용어 사전이 단순 표 형태라, 새 용어 추가 시 기존 용어와의 관계나 상위/하위 개념 구분이 어렵다.
- **버전 정책 미정의**: 0.1.0-draft 이후 어떤 기준으로 0.2.0, 1.0.0으로 버전이 올라가는지 정책이 없다.

### 구체적인 개선 제안

1. 문서 헤더 아래에 변경 이력 표를 추가하라.
   ```
   | 버전 | 날짜 | 변경자 | 변경 내용 |
   |------|------|--------|----------|
   | 0.1.0 | 2026-05-12 | BA | 초안 작성 |
   ```
2. 이해관계자 승인란을 추가하라. (역할, 이름, 서명/확인일 포함)
3. 문서 헤더 또는 말미에 "관련 문서" 섹션을 추가하고 ERD, API 명세 등의 파일 경로 또는 링크를 기재하라.
4. 버전 정책을 명시하라. (예: "주요 범위 변경 시 minor 버전 업, 이해관계자 최종 승인 시 1.0.0으로 확정")

### 전반적인 개선 방향

이 문서는 초안(draft) 수준으로서 도메인의 핵심 구조(엔티티, 관계, 비즈니스 규칙, 유스케이스, Bounded Context)를 빠짐없이 포함하려는 의도가 명확하다. 특히 명확성 측면에서 용어 사전과 ID 기반 규칙 체계는 팀 협업의 기반을 잘 마련하고 있다.

그러나 다음 세 가지 핵심 약점을 우선적으로 보완해야 한다.

**1순위: 검증 가능성 강화**
현재 문서만으로는 QA팀이 테스트를 설계하기 어렵다. 각 유스케이스에 수용 기준(Acceptance Criteria)을 추가하는 것이 가장 시급하다. 비즈니스 규칙도 "HTTP 상태 코드 + 결과 상태"까지 구체화해야 테스트 자동화가 가능해진다.

**2순위: 추적성 체계 구축**
문제(P)-비즈니스 규칙(BR)-유스케이스(UC) 간의 매핑 테이블이 없으면, 요구사항 변경 시 영향 범위를 파악하기 어렵다. 추적성 매트릭스(Traceability Matrix)를 별도 섹션 또는 첨부 문서로 작성하라.

**3순위: 완전성 보완**
비기능 요구사항(성능, 보안, 가용성), 입력값 유효성 제약, 예외 흐름 등 누락된 영역을 단계적으로 채워야 한다. 이 작업은 이해관계자 인터뷰 및 기술 아키텍처 결정 이후 수행하는 것이 효율적이다.

문서를 0.2.0 버전으로 업데이트할 때 위 3가지 개선 사항을 반영한다면, 종합 점수를 80점 이상으로 끌어올릴 수 있을 것으로 판단한다.
